## Test Prompts（3 个梯度）

### Test 1：最小可用性（新手能直接用吗？）
**Prompt:** "用Apple设计风格设计一个天气App的主界面"
**Expected Output Type:** 符合 Apple HIG 的 UI 方案（可直接用于设计参考）
**Must Include:**
- 信息层级清晰（主要天气信息 > 次要信息 > 详细数据，通过字号/粗细/颜色/间距区分）
- 典型 Apple 视觉元素（圆角 8-12px / SF Pro 字体族 / 系统蓝色调 / 卡片式布局 / 微妙阴影）
- 负空间的主动运用（不是"空白"而是"有意识的呼吸空间"）
- 至少描述 2-3 个关键交互（滑动/点击/长按的反馈）
- Dark Mode 考虑（不只是"支持深色"而是描述深色模式下的具体调整）
- 可访问性基础（对比度达标/触控区域足够大）

**Quality Bar:**
- ✅ Good: "顶部：城市名(SF Pro Regular 20pt) + 当前温度(SF Pro Bold 64pt系统蓝)+ 天气图标(96px SF Symbol)；中部：小时预报横向滚动卡片(圆角12px微妙阴影)；底部：7日预报列表；交互：下拉刷新模拟天气更新动画；Dark Mode：纯黑背景(#000000)卡片背景(#1C1C1E)"
- ❌ Bad: "一个干净的天气App，白色背景，圆角按钮，显示温度和天气预报，简洁大方" —— 只有形容词没有设计语言

### Test 2：核心能力（这个 skill 的独特价值在哪？）
**Prompt:** "评价这个UI设计，给出Apple风格的改进方案：[描述：一个电商App的商品详情页，顶部是大图轮播，下面是价格(红色大号)、标题、规格选择(横排按钮)、立即购买(大红按钮底部固定)、各种优惠标签散布在价格周围]"
**Expected Output Type:** Apple视角的设计评审（基于 HIG 原则的结构化批评）
**Must Include:**
- 逐项诊断（对照 Apple 四原则：直觉性/一致性/审美/效率）
- 具体问题识别（信息层级混乱/色彩滥用/视觉噪音/交互违和）
- 每个问题附 Apple 式解决方案（不是"改成蓝色"而是"使用语义化颜色系统：主操作用系统蓝/价格用标签灰/促销用次要橙色"）
- 改进前后的对比描述（让用户直观感受差异）
- 优先级排序（先改哪3件事提升最大）

**Quality Bar:**
- ✅ Good: "诊断：①层级崩塌——价格(红大号)和标题抢夺注意力违反'单一视觉焦点'原则②色彩暴力——红色价格+红色按钮+红色标签=色彩通胀③信息焦虑——优惠标签散布各处增加认知负荷。改进：①价格降权至标签灰Secondary Label②主CTA改为系统蓝填充③优惠信息收纳至折叠区④添加负空间'呼吸'⑤圆角统一为12px"
- ❌ Bad: "这个设计有点乱，建议简化一下颜色，按钮可以再大一点，整体还不错" —— 模糊且无用

### Test 3：大师级（真正的高手怎么用？）
**Prompt:** "做一个Apple风格的深色模式Dashboard（面向开发者的服务器监控面板）"
**Expected Output Type:** 系统级一致性设计（专业工具领域的 Apple 设计哲学应用）
**Must Include:**
- 专业工具 vs 消费级 App 的设计权衡（Apple设计语言在B端场景的适应性调整）
- 数据可视化的 Apple 式处理（图表样式/色彩编码/实时更新的动效/异常状态的呈现）
- 信息密度与简洁性的平衡（监控面板需要显示大量数据但不能显得拥挤）
- 深色模式的精细化（不是简单反色，而是重新设计光影层次：#000000 背景 / #1C1C1E 卡片 / #2C2C2E 分隔）
- 组件系统的描述（按钮/输入框/卡片/导航/通知的一致性规范）
- 动效设计（数据刷新/告警触发/页面切换的过渡动画，遵循Apple的"动效即功能"原则）
- 开发者友好性（标注设计令牌：色值/字号/间距/圆角的精确数值）

**Quality Bar:**
- ✅ Good: 定义设计系统Token(Colors/Typography/Spacing/Corner Radius) → Dashboard Grid(12列系统) → 数据组件库(Chart Card/Metric Card/Log Viewer/Alert Banner) → 深色模式专用调色板(非简单invert) → 动效规范(Ease-in-out 300ms for data update/Spring for alerts) → 可访问性(WCAG AA对比度)
- ❌ Bad: 只是画了一个"黑色的仪表盘"没有任何设计系统思维

## Failure Cases（2 个）

### Failure 1: "白色+圆角"= Apple 设计？
**When:** 用户要求 Apple 风格时
**What Goes Wrong:** 认为Apple设计=白色背景+大圆角+毛玻璃，忽略了Apple设计的核心是原则而非视觉特征
**Why:** 混淆了"表象"和"本质"——Apple的设计语言可以应用在任何配色和形态上（包括深色模式/工业风/甚至拟物时代的Apple也是Apple设计）
**Fix:** 回归四大原则（直觉性/一致性/审美/效率）——只要满足这四条，哪怕不用圆角也可以是Apple设计哲学

### Failure 2: 信息层级混乱（"简洁"≠"少"）
**When:** 处理信息密集的界面时
**What Goes Wrong:** 为了追求"Apple的简洁"而删除了必要信息，导致用户找不到需要的内容
**Why:** 误解了Apple的"简洁"——Apple的简洁是通过优秀的层级让复杂变得有序，而不是简单地减少内容
**Fix:** 用" Progressive Disclosure"（渐进式披露）——信息分层：默认显示核心→展开显示次要→深入显示详细，而不是一刀切地删除

## Style Fingerprint（风格指纹）

**Good Output Smells Like:**
- **克制的美**：每个像素都被需要，没有一个多余
- **系统思维**：不是在设计一个页面，而是在定义一个系统
- **人文关怀**：技术产品但有温度（动效/反馈/容错都考虑到人的感受）
- **精确的松弛感**：规则严格但看起来毫不费力（effortless precision）

**Bad Output Smells Like:**
- **极简主义抄袭**：只有"少"没有"好"（空白但不知所云）
- **iOS克隆**：机械照搬 iOS 掄件而不理解为什么这样设计
- **设计师自嗨**：好看但不好用（为了设计而设计）

## Before / After

**Before:** 用户做UI设计凭感觉，"简洁"等于"少放东西"，界面功能齐全但混乱
**After:** 用户掌握Apple设计原则的系统化应用，能在"美观"和"好用"之间找到最优平衡
**One-Line Delta:** 从"做得简单"到"让复杂变得清晰"
